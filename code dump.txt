I need a Streamlit application that acts as a lightweight CMS for an internal HTML site. The site is already built and deployed. The problem is efficiently updating specific sections when new data is available — without manually editing raw HTML.

## What the App Should Do

Build a Streamlit app (`site_editor.py`) that implements this workflow:

1. **Load** — User uploads or selects an existing HTML file
2. **Parse** — App auto-detects editable sections within the HTML
3. **Edit** — User edits individual sections using Markdown in text areas
4. **Preview** — User sees a full-page preview with edits applied (rendered in an iframe)
5. **Save** — User clicks a save button which creates a timestamped versioned copy AND updates the main file

## Technical Requirements

### Dependencies
- `streamlit` — UI framework
- `beautifulsoup4` — HTML parsing and reconstruction
- `markdown` — converting Markdown back to HTML (with `tables`, `fenced_code`, `nl2br` extensions)
- Python standard library only beyond these (no database, no external services)

### File Structure
```
$SITE_DIR/                     # configurable via env var, default ~/site_files
├── my_site.html               # the main working file
└── versions/
    ├── my_site_v20260327_143022.html
    ├── my_site_v20260326_180000.html
    └── ...
```

### Section Detection (Critical Logic)

The app must detect editable sections from arbitrary HTML using three strategies, tried in priority order:

**Strategy 1 — `<section>` tags:**
Find all `<section>` elements. Use the `id` attribute as the section identifier. Look for the first heading (h1-h6) inside each section to derive a display title. If no heading, derive the title from the id (replace underscores/hyphens with spaces, title case).

**Strategy 2 — `<div>` tags with `id` attributes:**
Only used if no `<section>` tags found. Find all `<div id="...">` elements. Skip generic wrapper IDs like "wrapper", "container", "app", "root", "main". Same heading/title logic as above.

**Strategy 3 — Heading-based splitting (fallback):**
Only used if neither of the above finds anything. Split the body content at `<h1>`, `<h2>`, or `<h3>` boundaries. Each heading starts a new section; content before the first heading becomes an "Introduction" section. The section ID is derived from the heading text (lowercase, non-alphanumeric replaced with underscores).

**Final fallback:** If nothing is detected at all, treat the entire body as a single editable section called "Full Page Content".

Each detected section should be stored as a dict:
```python
{
    "id": str,         # unique section identifier
    "title": str,      # human-readable display name
    "html": str,       # the inner HTML content of that section
    "tag_type": str,   # "section" | "div" | "heading_group" | "full"
    "tag_attrs": dict, # original tag attributes (for reconstruction)
}
```

### HTML-to-Markdown Conversion

When a section is opened for editing, convert its inner HTML to approximate Markdown so the user can edit comfortably. Handle at minimum:
- Headings (h1-h6) → `#` syntax
- Bold/strong → `**text**`
- Italic/em → `*text*`
- Links → `[text](url)`
- Unordered lists → `- item`
- Ordered lists → `1. item`
- Tables → pipe-delimited Markdown tables with header separator row
- Images → `![alt](src)`
- Paragraphs → separated by blank lines
- Line breaks → newlines

This does NOT need to be perfectly round-trippable. It's a convenience — users can always switch to raw HTML editing.

### HTML Reconstruction

When saving, the app must:
1. Convert each edited Markdown section back to HTML using `markdown.markdown()` with `tables`, `fenced_code`, and `nl2br` extensions
2. Replace ONLY the inner content of each detected section in the original HTML, preserving:
   - The `<head>` (styles, scripts, meta tags)
   - The overall document structure
   - Any sections that were NOT edited
   - All original tag attributes (classes, ids, data attributes)

For `<section>` and `<div>` tag types: find the tag in the parsed soup, clear its contents, and insert the new HTML.
For heading-based sections: use string replacement on the raw HTML (replacing the old inner HTML with the new).

### UI Layout

**Page config:** Wide layout, title "Site Section Editor"

**Sidebar:**
- File uploader (accepts .html, .htm)
- Dropdown to select from previously loaded files in SITE_DIR
- Version history list (most recent 10, sorted descending by timestamp)

**Main area — three tabs:**

1. **Edit Sections tab:**
   - One `st.expander` per detected section, showing the title and ID
   - Inside each expander: a `st.text_area` with the Markdown content (height ~300px)
   - Below the text area: two columns showing the raw Markdown and a rendered preview side by side

2. **Preview tab:**
   - Full rebuilt HTML rendered inside `st.components.v1.html()` with scrolling enabled and ~800px height
   - This gives full-fidelity preview including the original CSS

3. **Raw HTML tab:**
   - A large `st.text_area` with the fully rebuilt HTML
   - If the user manually edits here, those changes take priority when saving

**Save controls (below tabs):**
- A primary "Save New Version" button
- Caption explaining versioning behaviour
- On save: create timestamped file in versions/, update main file, show success + balloons
- Update session state so continued editing works on the new version

### Session State Management

Use `st.session_state` for:
- `current_file` — path to the active HTML file
- `original_html` — the full HTML content currently loaded
- `section_edits` — dict mapping section ID → current Markdown content
- `manual_raw_edit` — stores raw HTML if user edits in the Raw HTML tab (takes priority on save)

### Edge Cases to Handle
- HTML files with no clear section structure (use the fallback)
- Sections with no headings (derive title from ID)
- Unicode content
- Large files (don't crash, but no need for pagination)
- Empty sections (still show them as editable)
- User edits raw HTML directly — those edits should override Markdown-based edits on save
